2Topic normalization (write path)

How free-text topics from an SME interview become canonical topic_id values. Implemented in langgraph_sme_intake.normalize_topics_node.

1
LLM extracts free-text candidates from the SME's input
From the profile blob and interview turns, the LLM emits a list of free-form topic strings. Example output:
['CPT timeline', 'I-20 form', 'International student visa']
↓
2
For each candidate, normalize against topic_taxonomy.yaml
Cascade three matchers; first hit wins. Move on to the next matcher only on miss.
2a Exact match against topic.id or display_name? → map to that topic_id.
2b Alias hit on topic.aliases[]? → map to that topic_id. e.g. 'CPT timeline' → alias 'cpt application' → cpt
2c Embedding fuzzy match: cosine(candidate_embed, taxonomy_topic_embed) ≥ 0.85? → map to nearest topic_id. e.g. 'International student visa' → nearest is visa
2d None of the above → mark as unmatched; carry into state.unmatched_topics.
↓
3
Hits → canonical topic_id stored in knowledge_entries.topic_tag
Multi-tag (text[]). First element = primary. Saved during persist_node alongside the entry's question, answer, and embedding.
↓
4
Unmatched → admin_queue (source = sme_intake)
Admin reviews via scripts/admin_inbox.py and decides one of:
resolvedmap to existing topic_id (rename or merge)
needs_smepromote to a new topic and onboard an owner SME
dismissedout of scope for Thoth

3Hybrid retrieval pipeline (read path)

When a student asks a question. Implemented in langgraph_user_query.hybrid_retrieve_node + judge_node.

A
SQL hard filter
Strip rows that should never be returned, regardless of similarity. Topic filter applied only if the query mapped to known taxonomy ids.
WHERE status = 'approved' AND (topic_tag && query_topics)
↓
B
Parallel recall (top 20 each)
Two paths run independently. Each returns ranked candidates from the filtered pool.
Keyword pathtsvector
Postgres full-text on search_tsv (auto-generated from question + answer text).
Rank by ts_rank.
Vector pathpgvector
Cosine similarity over embedding column.
ORDER BY embedding <=> query_embedding
↓
C
RRF fusion → top 5
Reciprocal Rank Fusion combines the two paths.
score = 1/(60 + kw_rank) + 1/(60 + vec_rank)
Each candidate also carries flags: kw_hit (matched keyword) / vec_hit (matched vector). Used by the judge to detect off-topic queries.
↓
D
Confidence judge (decision tree, see Section 5)
Looks at top-1 score, score gap, kw_hit count, primary topic spread, SME spread.
Decides one of: answer / clarify / route_sme / route_admin.
↓
E
LLM grounded answer (or routing message)
For answer: synthesize from the top exposable entries; cite each fact inline with [entry_id]; UI strips citations from the visible bubble but keeps the cited version for the 'show sources' toggle.
For route_*: surface the SME's preferred channel from sme_profiles.routing_preferences; never invent an answer.

4Hierarchical fallback (cascade filtering)

When a tight query filter returns nothing useful, widen the lens. Try the most precise scope first; broaden only if recall fails.

Layer 3
Try by topic (tightest)
WHERE status='approved' AND topic_tag && ARRAY['cpt']   —  if top score is high, stop here.
↓ recall too low
Layer 2
Widen to SME
WHERE status='approved' AND sme_id = patrick   —  deferred for PoC. Production wires this in.
↓ recall still low
Layer 1
Widen to domain (broadest)
WHERE status='approved' AND domain = 'career_services'
↓ nothing matched
Admin
Route to admin (no SME owns this query)
Insert into admin_queue with source='user_query'; admin triages.
PoC scope: Only Layer 1 + Layer 3 are wired in hybrid_retrieve_node. Layer 2 (SME-level fallback) and Layer 4 (sub-topic granularity) are noted in data_schema.yaml as future work.

5Judge decision tree (when top-K spans multiple entries)

After RRF returns top 5, decide whether to answer, clarify, or route. Logic in langgraph_user_query.judge_node.

if not query_topics and not any(c.kw_hit for c in cands):
→ route_admin   // off-topic guard, no signal anywhere

elif top1.score < THRESHOLD_LOW:
owner = find_topic_owner(query_topics)
if owner: → route_sme   // case C: low confidence but topic has an owner
else: → route_admin

elif not top1.exposable:
→ route_sme (force)   // e.g. fee_waiver: case-by-case, never answer directly

elif len(top3.primary_topics) > 1 or len(top3.smes) > 1:
if score_gap < THRESHOLD_GAP:
→ clarify   // case B: multi-topic ambiguity, scores tight
else:
→ answer   // top1 dominates despite spread

else:
→ answer   // case A: confident, single SME, single topic
Case A Multiple high-score entries on the same topic / same SME → merge them, feed the LLM, get one answer with multiple [entry_id] citations.
Case B Multiple high-score entries on different primary topics or different SMEs and scores tight → ask the student a clarifying question to disambiguate before routing.
Case C All scores low (no entry confidently fits) → never fabricate. Look up topic owner from taxonomy; route to that SME, or to admin if none.
